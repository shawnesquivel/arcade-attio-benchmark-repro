Independent Replication Report · June 12, 2026

Reproducing "A 100× Token Difference": Arcade's Attio MCP Benchmark, Re-run From Scratch

Replication target: Hood, E. "MCP Toolkit Benchmark: Arcade.dev vs Composio, A 100x Token Difference," Arcade.dev, March 1, 2026 [1] and its companion repository [2]. All artifacts for this replication are public [4].

fresh Attio workspace, their seed script deterministic + agent-driven re-runs raw-API baseline tiktoken cl100k_base security claims out of scope

Abstract

Arcade.dev published a benchmark claiming its Attio MCP toolkit consumes roughly 100× fewer response tokens than Composio's. We replicated the study end-to-end: re-counted their committed raw data, seeded a fresh Attio workspace with their script, re-ran all 8 queries on both platforms — once deterministically with their exact parameters, once with independent small-model agents — and added a raw Attio API baseline. Their raw data is genuine (recount: exact match; live re-run: within 1% per platform) and a large gap is real. However, the 100.6× headline depends on three methodology choices that each favor Arcade: response formatting, unequal record limits, and selective call accounting. Correcting them yields 50–64×. Counterfactual transforms show ~8.5× would remain if Composio adopted field selection and metadata stripping; the residual is response-shape design, not data. The paper's security claims were out of scope for this replication.

1Background

On March 1, 2026, Arcade.dev published a benchmark comparing its Attio MCP toolkit against Composio's on 8 CRM queries over a seeded sandbox [1]:

"Across 8 CRM queries, Arcade consumed 7,426 tokens total while Composio consumed 747,083, which is greater than a 100x difference." — Hood (2026), introduction [1]

The paper open-sourced its raw responses, seed script, and eval prompts, and invited verification: "You can reproduce this yourself." This report is that reproduction, performed with no access to either vendor's internals and published with all artifacts [4]. The original article also makes security-related claims about Composio; per the scope of this work, those are not evaluated here.

2Claims Under Test

  • C1 — Headline ratio. "Composio provided 100.6x more response tokens than Arcade for the same 8 queries against the same workspace." [1]
  • C2 — Mechanism. "Composio returns every field on every record: all custom attributes, all built-in attributes, with no way to select specific fields" … "This is the raw Attio v2 API response passed through unmodified." [1]
  • C3 — Per-record cost. "A single company record expands to ~5,800 tokens in Composio's response format. In Arcade's, the same record with name selected is ~30 tokens." [1]
  • C4 — Context windows. "At Composio's rate, the same 8 queries consume 373% of a 200K window; the responses cannot fit in a single context." [1]
  • C5 — Cost at scale. "At the mid-market tier, the toolkit choice alone could be a quarter-million dollars a year. At enterprise scale, it could be $2M." [1]
  • C6 — Error recovery. "Query 07 (compound filter: Technology companies with 1K+ employees) required 4 tool calls in Composio vs 1 in Arcade." [1]

3Hypotheses

Fixed before running anything, each falsifiable against artifacts we would produce:

H1Data authenticity. Re-counting the repo's committed raw files with the paper's own script reproduces the published per-query table.Tests whether the published numbers match the published data.
H2Live replicability. Re-running the same tool calls against a freshly seeded workspace reproduces each platform's total within ±5%.Tests whether the result is reproducible, not a one-off artifact.
H3Methodological invariance. The ~100× ratio is stable when formatting, record limits, and call accounting are normalized.Tests whether the headline is a property of the tools or of the measurement choices.
H4Agent independence. The gap is determined by tool response design, not by the client or model driving the tools.Tested by replacing their Claude Code client with independent small-model subagents given only the natural-language prompts.
H5Irreducibility. The gap cannot be substantially closed without changing what data the API returns.Tested with counterfactual transforms of Composio's real responses (strip metadata, project fields).

4Method

Fig. 1 — Experimental pipeline. Editable Excalidraw source: diagrams/experiment-pipeline.excalidraw [4].

4.1 Materials

  • Workspace: fresh Attio sandbox seeded with the paper's scripts/seed_workspace.py — 50 companies, 100 people, 50 deals, 26 custom attributes. One undocumented dependency: the script assumes 6 custom deal stages ("Nurture" … "Closed Lost") already exist; all 50 deal creations fail with HTTP 400 on a default workspace. We created the stages via the Attio API, then re-ran the deals phase.
  • Toolkits: Arcade's Attio.QueryRecords (+ its required GetObjectSchema) via Arcade's execute API; Composio's ATTIO_LIST_COMPANIES / ATTIO_LIST_DEAL_RECORDS via Composio CLI. Both OAuth-connected to the same workspace. Snapshot date: June 12, 2026.
  • Measurement: identical to the paper — "tiktoken cl100k_base on complete raw JSON responses saved to disk" [1], using their count_tokens.py logic. We additionally save every payload in canonical minified and pretty-printed forms, so formatting can be controlled for.

4.2 Procedure

  • Arm A — deterministic replay: the exact tool calls recorded in their committed metadata.json, both platforms. Isolates the tools from any agent behavior.
  • Arm B — matched limits: same as A, but Arcade is given limit=500 on the queries where their run gave Composio 500 and let Arcade default to 25 — so both sides return the same 192 records.
  • Arm C — agent-driven: two independent small-model subagents (Composer), one per platform, given only the paper's 8 natural-language prompts — no parameter hints. Replaces their Claude Code + Sonnet 4.6 client to test H4.
  • Arm D — raw API baseline: the same 8 queries sent directly to Attio's v2 REST API with no toolkit, to locate each toolkit relative to the underlying API.
  • Decomposition: starting from the published 100.6×, change one variable at a time (formatting → limits → call accounting → counterfactual metadata-strip → counterfactual field projection) and recompute the ratio at each step.

5Assumptions & Limitations

  • A1 — Measurement boundary. Like the paper, we count saved response payloads only. Request tokens, MCP tool-schema definitions injected into context, system prompts, and model reasoning are excluded on both sides. The paper's Q07 narrative about context burned during retries is therefore not measurable from artifacts — theirs or ours.
  • A2 — Workspace shape. Ratios are specific to this seeded sandbox (26 custom attributes, mostly populated). Sparser or richer workspaces will scale Composio's per-record cost roughly linearly; Arcade's is insensitive to unselected fields.
  • A3 — Snapshot in time. Both products are point-in-time (June 12, 2026; paper: March 1, 2026). Either vendor can change response shapes at any time.
  • A4 — Client substitution. The paper used "Claude Code as the client with Claude Sonnet 4.6" [1]; we used deterministic scripts plus Composer subagents. Arms A and C converging on token-identical totals (§6.5) is evidence the client doesn't matter here, which is itself one of our findings.
  • A5 — Q6 date shift. Their Q6 cutoff ("deals created before March 2026") predates our seed date, which would match zero records. We shifted the cutoff to preserve their query's all-match semantics (50 of 50 deals). Their repo also labels Q6 "close date" while actually filtering created_at.
  • A6 — Pricing model. Cost figures use the paper's own assumption: Claude Sonnet 4.6 input at $3/M tokens, response payload tokens only.
  • A7 — Scope. The paper's security claims are not evaluated. Latency, correctness-of-results, and write operations are out of scope; all 8 queries returned correct, matching record sets on both platforms.

6.5 Agent-driven runs (H4)

Two independent small-model subagents, given only the 8 natural-language prompts, chose the same tools and equivalent parameters as the deterministic replay on both platforms and produced token-identical totals (Arcade 6,903; Composio 564,231, minified). The client and model are irrelevant to the outcome; tool response design fully determines it. H4 supported. Notably, the paper's Q07 story (C6) — 4 Composio calls with two failed filter attempts — did not recur: our agent wrote a valid compound filter on the first try, having seen valid option titles in earlier query responses. Cold-start retry friction is plausible but contingent on agent state, not structural to the toolkit.

8Discussion — Claim-by-Claim Scorecard

  • HOLDS C2 — Composio is a passthrough of Attio's API. Confirmed quantitatively: 98.6% of the raw-API token count on identical queries. The paper's phrase "raw Attio v2 API response passed through unmodified" is the single most accurate line in it — and it also reframes the problem: the verbosity is Attio's response format, relayed rather than created. The fix is scoped engineering (strip wrapper → 32.8×, field selection → 8.5×, flatten → near-parity), not a fundamental property.
  • HOLDS C3 — Per-record cost. ~5,670 tokens per company record as committed (their "~5,800") vs ~34 for Arcade ("~30"). Verified (§6.6).
  • HOLDS (directionally) C4 — Context windows. Their "373% of a 200K window" includes the formatting penalty; the fair number is 282%. The conclusion survives: a full 8-query Composio run cannot fit in a 200K context. Arcade's, schema calls included, uses 5.6%.
  • INFLATED C1 — "100×" needed methodology help. Composio's files were saved pretty-printed (+~33% tokens vs minified), Arcade's semi-minified; their own run requested limit=500 from Composio while Arcade defaulted to 25 (192 vs 156 records); Arcade's required schema-discovery calls were saved but excluded from totals. Corrected: 50–64×. Large and real — but every uncorrected choice pointed the same direction.
  • INFLATED C5 — Cost at scale. Recomputed at the paper's own pricing ($3/M input) with fair accounting: a mid-market tier saves ~$187K/year, not "a quarter-million" — same order of magnitude, ~25% lower. Per 8-query run: $1.69 vs $0.034.
  • NOT REPRODUCED C6 — Q07 error recovery. Our agent-driven Composio run completed Q07 in one call (§6.5). The 4-call anecdote depends on agent state and is unverifiable from their artifacts (A1); it should not be read as a structural toolkit property. We also note the asymmetry: Composio's one-off schema lookup is narrated as overhead while Arcade's mandatory schema calls are excluded from totals.
  • MISLEADING Per-query framing. The headline-making per-query ratios (160×, 161×) are the queries where formatting and limit artifacts stack. The fair per-query range is 19×–125×; the single-record query that isolates pure per-record overhead (Q08) is 19×.

9Conclusion

100.6×
Published headline. Genuine data (H1, H2 supported) measured with a methodology whose three uncorrected choices — formatting, limits, call accounting — all favored the vendor publishing it (H3 rejected).
50–64×
The defensible number. Same records on both sides, both payloads minified: 63.6×. Counting Arcade's required schema-discovery calls: 50.0×. Still a real, large gap — Composio relays Attio's verbose API format wholesale.
8.5×
The engineering view. What would remain if Composio stripped Attio's per-value metadata and projected requested fields (H5 rejected). The residual is response flattening — design, not data.

The benchmark's direction is right and its data is honest; its magnitude is partly a measurement artifact, and its framing presents a fixable integration choice as an intrinsic gap. Read as engineering guidance rather than marketing: response shape is the product. A toolkit that projects fields and strips metadata turns the same API into ~2% of the tokens — and that decision, not the model, the client, or the agent, determined every number in this study.
