Why The Gap Exists — Same Record, Both Tools

One seeded company ("Airbnb") as each tool returns it. This is the entire story:

// Arcade Attio.QueryRecords — 36 tokens
{"name": "Airbnb",
 "record_id": "0b1f3334-…"}
// Composio ATTIO_LIST_COMPANIES — 2,818 tokens
// 17 attributes, EACH wrapped like this:
"name": [{
  "active_from": "2026-06-12T20:34:14.220000000Z",
  "active_until": null,
  "attribute_type": "text",
  "created_by_actor": {"id": null, "type": "system"},
  "value": "Airbnb"
}]
  1. Composio is a passthrough. Its responses are 98.6% the size of the raw Attio v2 API for identical queries (564,231 vs 572,293 tokens). It adds nothing — and removes nothing.
  2. Arcade is a post-processor. Its QueryRecords projects requested fields, strips Attio's temporal/actor metadata, and flattens values. That's a real product difference, and it costs a schema-discovery round-trip first.
  3. The red keys are the bloat. Attio wraps every value of every attribute with history metadata. No CRM query in this benchmark needed any of it.

Claim-by-Claim Scorecard

  • HOLDS Their raw data is real. We re-counted their committed files (7,426 / 747,083 — exact match) and re-ran every call on a fresh workspace, landing within 1% per platform.
  • HOLDS Composio's Attio responses are enormously verbose. Even the fairest framing is ~50–64× on this workload. A single record costs ~1,7–2,8K tokens. A full 8-query run (564K tokens) cannot fit in a 200K context window; Arcade's fits in ~6%.
  • INFLATED "~100×" needed methodology help. Their Composio files were saved pretty-printed (+25% tokens), Arcade's were semi-minified. Their own run gave Composio limit 500 but let Arcade default to 25 — comparing 192 records vs 156.
  • INFLATED Selective accounting. Arcade's flow required schema-discovery calls (they saved them: 1,514 tokens committed, 2,412 in our agent run) — excluded from their totals. Including them: 50×, not 100×.
  • MISLEADING "160× on query 1" framing. The worst per-query ratios are formatting + limit artifacts. Fair per-query range is 19×–125×, and the single-record case (q8) — the floor that isolates per-record overhead — is 19×.
  • FRAMING "Composio sends garbage" → actually "Composio sends Attio." The verbosity is Attio's native API format passed through. That's still a worse agent experience — but the fix is well-scoped engineering (strip wrapper → 33×, field selection → 8.5×, flatten → parity), not a fundamental flaw.

Repro Notes & Friction We Hit

  1. Seed script assumes manual setup it never mentions. All 50 deal creations 400-failed on a fresh workspace — stages like "Nurture" don't exist by default. We created the 6 statuses via API and re-ran. Their published repro steps skip this.
  2. Query 6 is mislabeled in their repo. "Deals by close date" actually filters created_at (record creation time), not close date — it matches every deal in the workspace regardless. We shifted the cutoff date to preserve identical all-match semantics on our seed date.
  3. Their Arcade truncation is visible in their own files. Queries 3, 5, 6, 7 show has_more: true at 25 records while Composio returned 30/29/50/27. The headline compares unequal workloads.
  1. Small-model agents converged exactly. Two Composer subagents, given only the natural-language queries, picked the same tools as the original run on both platforms and produced byte-identical token totals (6,903 / 564,231 minified) — the benchmark is robust to agent choice; tool design fully determines the outcome.
  2. Cost framing at GPT-4o input pricing ($2.50/M): one 8-query run = $1.41 via Composio passthrough vs $0.028 via Arcade (schema calls included). At 1,000 runs/day that's $1,411 vs $28 — the gap is real money, whatever the exact multiplier.
  3. Security claims were out of scope per the brief; nothing here evaluates them.
